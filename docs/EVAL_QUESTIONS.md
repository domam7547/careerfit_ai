## 프로젝트 전체
- 이 서비스가 해결하는 문제는 무엇인가? 
A: 대학교 고학년학생, 졸업생 및 취업준비생
- RAG를 사용하지 않으면 어떤 문제가 생기는가?
취업공고 데이터가 프롬프트에 들어가지 않아서, 범용 인공지능이 실제 공고를 근거로 요약하고 추천하지 못함. -> 일반론적인 이야기만 하게됨.
- 가장 어려웠던 구현은 무엇이고, 어떻게 해결했는가?
ChromaDB / ONNX 실행 환경 문제

  - 문제: RAG 검색을 실행할 때 ChromaDB 내부 임베딩 단계에서 onnxruntime이 임시 작업 디렉터리를 만들지 못해서 검색이 실패했습니다.
  - 영향: /analyze 전체 흐름이 중간에서 멈췄고, Gemini 성능 문제가 아니라 검색 자체가 막히는 상황이었습니다.
  - 해결 방향:
      - 처음엔 Chroma telemetry 경고가 원인처럼 보였지만, 로그를 다시 분리해서 보니 진짜 원인은 ONNX 임시 디렉터리 제약이었습니다.
      - 즉, 코드 버그로 단정하지 않고 실행 환경 문제와 애플리케이션 문제를 분리해서 진단했습니다.

  - 이걸 어려웠다고 말할 수 있는 이유:
      - 겉으로는 RAG가 실패하는데 실제 원인은 모델 품질도, API 키도, 라우터도 아니었음
      - “무엇이 진짜 장애 원인인가”를 로그로 좁히는 과정이 필요했음

## 데이터 관련
- 왜 이 데이터를 선택했는가? 
서비스 이용자들이 가장 궁금해하는 데이터이기 때문이다.
- 데이터의 어떤 부분이 신뢰할 수 없을 수 있는가?
정확한 출처에 관한 링크가 없기 때문에 데이터 제공자만 신뢰할 수 밖에 없다.
- SQLite와 ChromaDB에 같은 데이터를 저장하는 이유는 무엇인가? 
  - SQLite
      - 구조화된 데이터를 표 형태로 저장
      - 조건 검색, 필터링, 정렬, 관리에 강함
      - 예: 회사명, 직무, 마감일, 공모전/채용 구분 같은 컬럼 기반 조회

  - ChromaDB
      - 문장 의미 기반 검색에 강함
      - 사용자의 자연어 질문과 비슷한 공고를 찾는 데 사용
      - 예: “Python과 SQL이 필요한 데이터 분석 공고 찾아줘” 같은 문장을 벡터 검색 
- 결측치와 중복을 어떻게 처리했는가?
1. 핵심 컬럼을 기준으로 결측치를 확인했다
2. 서비스에 의미 없는 결측 행은 제거했다
3. 중복 행은 제거했다
4. 정리된 데이터만 SQLite와 RAG 문서로 넘겼다

## RAG 관련
- RAG 구조에서 ChromaDB는 어떤 역할을 하는가?
ChromaDB는 사용자의 자연어 질문과 의미적으로 비슷한 취업공고 문서를 찾는 벡터 검색 저장소 역할을 한다. 즉, LLM이 일반 지식만으로 답하지 않고 실제 공고 데이터를 근거로 답하도록 연결하는 검색 계층.
- n_results 값을 높이면 어떤 트레이드오프가 생기는가?
더 많은 공고를 참고할 수 있어서 관련 정보를 놓칠 가능성은 줄어든다. 대신 관련성이 낮은 문서가 섞일 수 있고, 프롬프트가 길어져 응답 속도와 답변 집중도가 떨어질 수 있음
- 검색 결과 품질을 어떻게 검증했는가?
같은 질문을 넣었을 때 사용자 의도에 맞는 공고가 반환되는지 직접 확인했다. 또 `/analyze` 응답에 `sources`를 함께 포함해서, 실제로 어떤 공고가 답변 근거로 사용됐는지도 검증했음
- Hallucination이 무엇이며, RAG가 이를 어떻게 줄이는가?
Hallucination은 모델이 실제 데이터에 없는 내용을 그럴듯하게 만들어내는 현상이다. RAG는 검색된 공고 데이터를 프롬프트에 함께 넣어 모델이 실제 근거를 바탕으로 답하게 하므로, 이런 환각 가능성을 줄여줌.

## API 관련
- /analyze 엔드포인트가 POST를 사용하는 이유는 무엇인가?
전공, 스킬 목록, 관심 직무처럼 구조화된 입력 데이터를 JSON body로 받아야 하기 때문이다. 의미적으로도 단순 조회가 아니라 분석 요청에 가깝기 때문에 POST가 적절함.
- 응답 JSON에 sources를 포함시킨 이유는 무엇인가?
AI 답변의 근거가 된 공고를 함께 보여주기 위해서다. 사용자는 왜 이런 추천이 나왔는지 확인할 수 있고, 발표나 면접에서도 근거 기반 추천 구조라고 설명할 수 있음.
- /health 엔드포인트가 왜 필요한가?
서버가 정상적으로 살아 있는지 가장 빠르게 확인하기 위한 상태 점검용 엔드포인트다. 로컬 개발, Docker 실행, Render 배포 후 확인까지 모두 이 경로로 점검할 수 있음.

## 보안 관련
- .env 파일을 GitHub에 올리면 어떤 일이 생기는가?
Gemini API 키 같은 민감한 값이 외부에 노출될 수 있고, 무단 사용이나 과금 문제가 생길 수 있다. 그래서 실제 `.env`는 Git에 올리지 않고, 필요한 변수 이름만 담은 `.env.example`만 공유함.
- .env와 .env.example의 차이는 무엇인가?
`.env`는 실제 실행에 사용하는 비밀값 파일이고, `.env.example`은 어떤 환경변수가 필요한지 보여주는 템플릿 파일이다. 즉, `.env`는 실제 값, `.env.example`은 예시

## Docker 관련
- Docker를 사용하는 이유는 무엇인가?
로컬과 배포 환경 차이를 줄이고, 같은 실행 환경을 재현하기 위해서다. 특히 Render에 배포할 때 개발 환경과 비슷한 방식으로 안정적으로 실행할 수 있다는 장점이 있음.
- Dockerfile에서 EXPOSE와 CMD의 역할은 무엇인가?
`EXPOSE`는 컨테이너가 사용할 포트를 문서적으로 알려주는 역할이고, `CMD`는 컨테이너가 시작될 때 기본으로 실행할 명령을 지정하는 역할임.
